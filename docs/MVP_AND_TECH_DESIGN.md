# MVP 范围 & 技术设计

## 一、MVP 版本

### 基础设定
- **目标用户**：70/100 分段高中生
- **学科**：高中数学，仅函数 + 导数两章
- **部署**：Web 应用
- **账号**：无登录，单用户；浏览器本地存储；保留历史，用户可回看历次诊断和答题
- **LLM**：Kimi K2.5 全栈使用
  - 读用户上传的错题（PDF / 图片 / 文字 / txt）
  - 基于学科知识库 + 错题生成诊断报告
  - 上传母题时一次性预生成通用题目思考路径并存档

### 内容准备（产品侧，PDF 交付）
- 函数 + 导数两章的结构化知识点 tag 清单 + 目录（不含关系说明，关系由 LLM 推测）
- 母题 100 道以内 + 每道的母题解析
- 1–2 个真实学生的错题样本（首次跑通用）

### 核心流程
1. **用户上传错题**：手动文字录入 / PDF / 图片 / txt；下限 5 题，上限不限；无需标注。
2. **系统生成诊断报告**
   - 三句话：
     1. 薄弱知识点是 X、Y、Z
     2. 它们在整个体系中的结构关系（LLM 推测）
     3. 基于结构关系，当前应着重强化的知识点
   - 推荐 5 道练习题（按薄弱 tag 从母题池中召回）
3. **用户答题**：纯选择题，一题一答；提交立刻显示标答 + 通用思考路径；系统自动判对错。
4. 5 题答完，用户可选"再来 5 题"或结束本次。

### 已砍（v2 再考虑）
- 知识图谱可视化
- 本周 / 2 小时目标
- 思维模式分析
- 错题循环回流再诊断
- 进步追踪
- 多学科

## 二、技术设计

### 开工前两个选择

#### 1. 技术栈（vibe coding 越简单越好）

**推荐：Next.js 14 + TypeScript + Tailwind + Vercel**

- 前后端一体，Kimi API key 放在 server 端不暴露
- AI 编程工具对 Next.js 支持最好

#### 2. 部署
Vercel（免费档够 MVP 用）

---

### Phase 0：内容准备（产品侧，与开发并行）

- **P0-1** 知识点 tag 清单 + 目录 PDF（每点带唯一 tag，如 `func.def`、`func.monotonic`；目录体现章节层级；不写关系）
- **P0-2** ≤100 道母题 PDF（每道：题干 + 4 选项 + 标答 + 解析 + 1–3 个知识点 tag）
- **P0-3** 1–2 个真实学生错题样本（任意形式）

---

### Phase 1：项目脚手架 + 内容入库

- **P1-1** Next.js 14 + TypeScript + Tailwind 初始化；接入 Kimi API（OpenAI SDK 兼容，base URL 指向 Moonshot）；环境变量 `MOONSHOT_API_KEY`
- **P1-2** 上传 P0-1 PDF，调 Kimi 多模态解析 → 输出 `data/knowledge.json`（每个 tag：`id`、`name`、`chapter`、`parent_tag`）
- **P1-3** 上传 P0-2 PDF，调 Kimi 多模态逐题解析 + 对每道题再调一次 Kimi 生成"通用思考路径" → 输出 `data/problems.json`（每题：`id`、题干、选项、答案、解析、思考路径、tag 数组）

**完成标志**：两个 JSON 落地，浏览器里 fetch 出来正常。

---

### Phase 2：错题上传 + 诊断

- **P2-1** 上传页 UI：文字粘贴框 / PDF / 图片 / txt 四种入口；错题数量 < 5 不允许提交
- **P2-2** Kimi 多模态识别 → 输出结构化错题列表（每道题：题干、用户答案、正确答案、涉及知识点 tag）
- **P2-3** 诊断报告：输入错题 tag 分布 + `knowledge.json`，调 Kimi 输出三句话（薄弱点 / 结构关系 / 应着重强化的部分）
- **P2-4** 从 `problems.json` 按薄弱 tag 召回 5 道，按薄弱度排序

**完成标志**：上传错题样本 → 三句话诊断 + 5 道推荐题。

---

### Phase 3：答题交互

- **P3-1** 一题一答 UI（题干 + 4 选项 + 提交）
- **P3-2** 提交后立刻显示对错 + 标答 + 母题解析 + 通用思考路径 + "下一题"
- **P3-3** 5 题完结：显示本轮对错统计 + "再来 5 题"（从同一诊断的薄弱 tag 池召回，去重上一轮已答的题）+ "结束"

**完成标志**：完整跑完一轮 5 题。

---

### Phase 4：本地持久化 + 历史回看

- **P4-1** IndexedDB 设计两张表：
  - `diagnoses`（错题输入 + 三句话报告 + 推荐题 ID）
  - `practice_sessions`（每轮答题记录）
- **P4-2** 历史页：按时间倒序列出历次诊断，点进去看当时的报告 + 答题记录

**完成标志**：刷新页面历史仍在，可回看。

---

### Phase 5：真实数据跑通 + 修 bug

- **P5-1** 用 P0-3 的真实错题走完整流程
- **P5-2** 列出识别错、诊断不准、UX 卡顿的点逐个修
- **P5-3** 给 1–2 个真实学生用一次，收一轮反馈

**完成标志**：真实学生愿意继续用。
